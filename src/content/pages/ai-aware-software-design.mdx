<p>Preface: I've been building <a href="https://posthaste.theor.net/?ref=ghost.theor.net">Posthaste</a> since March. It's a modern, smart, fast mail workstation blah-blah-blah: check it out. It is my first big application since LLM dawn. Along the way I've rethought what mail should look like in an era of AI agents. Yes, this blog post will coincidentally advertise Posthaste further. </p><p>LLMs equip everyone with a pocket "computer guy". It would be naive to think this doesn't change things for software developers. I haven't seen anyone make a convincing argument for what exactly this means; I haven't seen, embarrassingly, many apps adopting a new paradigm either. Hopefully this blog post helps the situation a bit. </p><h2 id="1-users-have-a-mediocre-tireless-coder">1. Users have a mediocre, tireless coder</h2><h3 id="11-make-your-application-text-editable">1.1. Make your application text-editable</h3><p>Put everything in a .toml, .json, .md or other plain text format. This will make it easier for users to debug and reconfigure your app, and you can expose arbitrary technical knobs without sacrificing usability. It doesn't really matter if your schema will wind up complex, but make sure to provide documentation.</p><p>Put files in a predictable place to save your users some tokens. Don't confuse the agents, and don't create situations susceptible to hallucination. Put links to where each section of your config, registry and state is documented. </p><div data-callout="" data-callout-type="info"><div data-callout-title=""><span data-callout-icon="info"><svg xmlns="http://www.w3.org/2000/svg" width="16" height="16" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-info" aria-hidden="true"><circle cx="12" cy="12" r="10"></circle><path d="M12 16v-4"></path><path d="M12 8h.01"></path></svg></span>Info</div><div data-callout-body="">Posthaste puts account registry in a .toml file, instead of using a more convenient SQLite. It points agents to documentation at <a href="https://theoryzhenkov.github.io/posthaste/?ref=ghost.theor.net">https://theoryzhenkov.github.io/posthaste</a>. </div></div><p><strong>Ex: </strong>Though not designed with LLMs in mind, git, notmuch, Maildir, Tailscale are good implementations of this pattern. It is not a coincidence that these are mostly developer tools. </p><h3 id="12-provide-a-localhost-exposed-api">1.2. Provide a localhost-exposed API</h3><p>How cool would it be to auto-add "zero inbox" event to your calendar when your inbound mail overflows a limit? Or display important mail on your TV? Or trigger an agent to do something on your VM when a mail with a specific tag arrives?</p><p>If I have my whole API stream on localhost, you can subscribe to listen to arbitrary events, fetch data by trigger, and so forth. You wouldn't bother, but now it is mostly your agent that has to debug and read my API. Cool!</p><div data-callout="" data-callout-type="danger"><div data-callout-title=""><span data-callout-icon="triangle-alert"><svg xmlns="http://www.w3.org/2000/svg" width="16" height="16" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-triangle-alert" aria-hidden="true"><path d="m21.73 18-8-14a2 2 0 0 0-3.48 0l-8 14A2 2 0 0 0 4 21h16a2 2 0 0 0 1.73-3"></path><path d="M12 9v4"></path><path d="M12 17h.01"></path></svg></span>Danger</div><div data-callout-body="">Just don't forget to require authentication, and enable same origin by default, otherwise any website from your browser will be able to interact with your app with privileged rights. No, no users were compromised due to Posthaste, I caught this early.</div></div><p><strong>Ex:</strong> Syncthing is almost a line-for-line match for what I described. It exposes a REST API on <code>localhost:8384</code>; you long-poll <code>GET /rest/events</code>, and create a subscription by adding an <code>events=TypeA,TypeB</code> filter for only the event types you want, plus a <code>since=&lt;lastSeenID&gt;</code> cursor so you resume exactly where you left off. Also look at Home Assistant. </p><h4 id="121-write-an-mcp-server">1.2.1. Write an MCP server</h4><p>If you expose the API, might as well make it easy for agents to work with it in a structured way. MCP has its own problems, including security ones, so users need to be careful — but for you it introduces no new vulnerability surfaces, so it is essentially free. </p><h3 id="13-make-your-software-modular">1.3. Make your software modular</h3><p>Modularity always sounded nice. Now, since more users can exploit it, and it is cheaper for you, it is also worth it. <strong>Ex:</strong> Posthaste splits in three: a backend (talks to providers, holds canonical data), a runtime (holds client state), and a client (renders). They only ever talk over an abstract transport, which can be HTTP, WebSocket, loopback, whatever. </p><p>This is how LSPs already work. The design goal is precisely the same: the language server is transport-agnostic — it adheres to the protocol and does not know about the transport layer; a separate launcher component is what's aware of the communication channel. That channel can be stdio, sockets, named pipes, or node IPC, and because of that transport-independence the server can run locally or in the cloud while you develop on your laptop.</p><h4 id="131-deployment-modes">1.3.1. Deployment modes</h4><p>Because the parts don't care how they're wired, I assemble them differently at CD time, <strong>Ex:</strong> </p><ul><li>Split: headless backend on the server; an app with bundled runtime + client connects to it over loopback. When you run the app on several devices, you've got shared smart mailboxes, filters, drafts, and backend-level cache.</li><li>Headless: backend and runtime bundled, no client, runs as a daemon on your machine.</li><li>Bundled: everything together, for a classic application experience.</li><li>etc.</li></ul><p><strong>Ex:</strong> The classic example is mpd (Music Player Daemon): a headless daemon holds the canonical library and playback state, and any number of thin clients (ncmpcpp, mpc, phone apps) connect over a socket or TCP to control the <em>same</em> daemon from different devices.</p><h4 id="132-swappable-parts">1.3.2. Swappable parts</h4><p>The same seams let people bring their own components. Missing a feature in the client? Ask codex to fork it and add the feature — I already did the hard part in the runtime underneath, so you point the fork at it and you're done. </p><div data-callout="" data-callout-type="info"><div data-callout-title=""><span data-callout-icon="info"><svg xmlns="http://www.w3.org/2000/svg" width="16" height="16" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" class="lucide lucide-info" aria-hidden="true"><circle cx="12" cy="12" r="10"></circle><path d="M12 16v-4"></path><path d="M12 8h.01"></path></svg></span>Info</div><div data-callout-body="">In the future Posthaste will take JS extensions to the frontend directly, so you won't even need to rebuild.</div></div><p><strong>Ex:</strong> notmuch. The library does the indexing and search, and the frontends are independent — emacs, vim, and mutt integrations, aerc, alot (Python/urwid), astroid (C++/GTK), bower (Mercury), meli (Rust), and web frontends written in Clojure, PHP, and JS. </p><h2 id="2-you-have-a-tireless-mediocre-coder">2.
